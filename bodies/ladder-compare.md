# ladder-compare

A를 배웠으면, B와 C가 왜 다른지 알아야 진짜 이해한 것입니다.
단순 스펙 비교가 아니라 **설계 의도의 차이**를 인과적으로 설명합니다.

## 사용 방법

```
/ladder-compare $TOPIC [$ALTERNATIVES]
```

- `$TOPIC`: 기준이 되는 기술/개념. 예: "TCP"
- `$ALTERNATIVES`: 비교 대상 (선택). 예: "UDP, QUIC". 생략 시 주요 대안을 자동 선택.

---

## 출력 원칙

1. **왜 이것이 선택되었는가 먼저**: 비교표 전에 각 기술이 어떤 문제를 풀기 위해 만들어졌는지 설명.

2. **비교 기준을 명시**: 임의로 비교하지 않고, 비교 축(성능/안정성/복잡도/적합 상황)을 먼저 정한다.

3. **출력 구조**:
   ```
   ## 설계 의도 비교
   - A는 [어떤 문제]를 풀기 위해 [어떤 트레이드오프]를 선택했다.
   - B는 [어떤 문제]를 풀기 위해 [다른 트레이드오프]를 선택했다.

   ## 비교표
   | 항목        | A   | B   | C   |
   |-------------|-----|-----|-----|
   | 연결 설정   | O   | X   | △   |
   | 순서 보장   | O   | X   | O   |
   | ...         |     |     |     |

   ## 언제 어떤 걸 쓰는가
   - A를 선택할 때: ...
   - B를 선택할 때: ...
   - 절대 섞으면 안 되는 경우: ...

   ## 역사적 맥락 (선택)
   왜 A가 먼저 나왔고 B가 나중에 나왔는지.
   ```

4. **한계 명시**: "이 비교는 [어떤 조건] 기준이며, [다른 조건]에서는 결론이 달라질 수 있다."
