# 기능 검증 문서 작성

## Overview

이 스킬은 **검증 문서(verification document) 자체가 어떻게 구성되어야 하는지**를 정의한다. 검증 문서는 아래 3가지 품질 기준을 만족해야 한다:

1. **독립성** — 문서만 보고 누구나 동일한 테스트를 수행할 수 있어야 함
2. **재현성** — 동일한 데이터/절차로 반복 가능해야 함
3. **구체성** — 기대 결과가 모호하지 않고 사람이 육안으로 확인 가능해야 함

## 문서 구조

검증 문서는 아래 섹션 순서를 따른다:

```
## 1. 검증 개요
## 2. 사전 준비
## 3. 데이터 준비 (SQL + 존재 이유)
## 4. 시나리오 목록
## 5. 시나리오 상세
## 6. 데이터 정리
## 7. 체크리스트
```

## 섹션별 작성 기준

### 1. 검증 개요

- 검증 대상 기능 설명 (1~2문장)
- 관련 Jira 티켓 또는 요구사항 링크
- 사전 배포 상태 (어느 환경에 배포되어야 하는지)

## 2. 사전 준비

검증에 필요한 환경/계정 정보를 간략히 적는다.
- 대상 환경 URL (예: `https://dev-xxx.example.com`)
- 리포트 설정 화면 접근 권한이 있는 테스트 계정
- Dev DB 접속 정보 (SQL 실행용)

> API 인증 방식(토큰/쿠키 등), curl 명령어는 시나리오에 포함하지 않는다.
> Feature는 UI로 검증하고, 필요한 데이터는 SQL로 준비한다.

### 3. 데이터 준비 (SQL + 존재 이유)

각 SQL 블록마다 데이터가 왜 필요한지를 항목화해서 붙인다.

```
**필요한 이유:**
- [데이터A]: 검증 대상 시나리오에서 기본 역할
- [데이터B]: 다른 데이터와의 대비를 통해 차이를 드러냄
- [데이터C]: 엣지 케이스 — 이 데이터가 없으면 검증 불가
```

**SQL 작성 규칙:**
- **과거 날짜만 사용.** 조회/리포트/다운로드는 미래일 불가능
- **실제 Dev DB 포맷과 동일하게 작성.** UUID 길이, datetime 포맷, ENUM 값 일치
- **테스트 식별자 prefix** 사용. 정리 SQL이 실제 데이터를 건드리지 않도록

### 4. 시나리오 목록

문서 초반에 시나리오 목록을 표로 정리:

| 번호 | 시나리오 | 검증 항목 | 검증 방식 |
|------|---------|-----------|----------|
| 1 | 옵션 OFF | 옵션 Off면 변화 없음 확인 | UI |
| 2 | 옵션 ON + 단일 선택 | 단일 유형별 컬럼 생성 확인 | UI |
| 3 | 옵션 ON + 복수 선택 | 복수 유형별 컬럼 + 합산 일관성 | UI |

### 5. 시나리오 상세

각 시나리오는 아래 2단계로 구성한다. 실행 절차와 기대 결과를 분리하지 않고 **검증 순서** 안에 순서대로 나열한다. 각 단계의 기대 결과는 그 단계 다음에 바로 적는다.

#### 전제 조건
- Feature 옵션 설정값 (On/Off, 선택값)
- 데이터 준비 섹션(3번)에서 준비한 데이터 중 어떤 데이터를 사용하는지
- 리포트 설정/생성 방법 (SQL 대신 **UI 조작**으로 설명. SQL이 꼭 필요한 경우만 최소한으로)
    - "리포트 설정 화면에서 생성" — 필드/옵션 설명
    - 단순 UPDATE 1줄이면 전제 조건에 인라인으로 포함 가능
    - 대량 INSERT는 여기에 두지 않는다. 데이터 준비 섹션(3번)에만 둔다

#### 검증 순서

절차와 기대 결과를 섞지 않고 **순서대로 나열**한다.

아래 형식을 권장:

```
1. 리포트 설정 화면에서 [리포트명] 선택 후 [옵션] 설정
   → 엑셀 다운로드 실행

2. 엑셀 요약 시트 확인
   - **열 구조:** 어떤 열이 생기고 사라지는지
     - 구체적인 열 이름과 개수 (예: 휴일 근로시간 2열 + "휴무" 휴일 근로시간 2열)
   - **값:** 구체적인 숫자, 텍스트, 포맷
     - (표로 정리 권장: 비교할 대상별로 행/열 구성)

3. 합계/집계 일관성 확인 (있을 경우)
   - 총합 = 항목별 합계 일치 여부를 구체적인 값으로 제시
```

**검증 순서 작성 원칙:**
- 실행 절차와 기대 결과를 분리하지 말고, 절차를 적고 바로 그 결과를 적는다
- 코드 내부값(`getter 반환값`, `resolveXxx() 조건`) 대신 **화면/엑셀에 보이는 값**을 기술
- "~여야 함" 대신 구체적인 값 제시 (예: "420분", "2개 열 추가")
- 합계/집계가 있다면 일관성 검증 포함 (예: "휴일 근로시간 합계 600분 = 420+180 일치")

### 6. 데이터 정리

- 각 테이블별 DELETE SQL
- 테스트 식별자 prefix로만 필터링 (실제 데이터 보호)
- 실행 순서 (FK 관계 고려)

### 7. 체크리스트

문서 완성 후 아래 항목 자가 검증:

- [ ] 모든 날짜가 과거 기준인가?
- [ ] DB 포맷이 실제 Dev DB와 일치하는가?
- [ ] 각 데이터 블록에 "필요한 이유:" 항목이 있는가?
- [ ] 시나리오의 데이터 준비(SQL)가 데이터 준비 섹션(3번)에만 있고, 시나리오 내에는 없는가?
- [ ] 시나리오 내 전제 조건에 대량 INSERT SQL이 포함되어 있지 않은가?
- [ ] 각 시나리오가 「전제 조건 → 검증 순서」 2단계로 구성되었는가?
- [ ] 검증 순서에 실행 절차와 기대 결과가 분리되지 않고 순서대로 나열되었는가?
- [ ] 시나리오가 코드 내부값이 아닌 사용자 화면/응답 기준인가?
- [ ] 각 시나리오의 기대 결과가 구체적인 값으로 명시되었는가?
- [ ] 합계/집계 일관성 검증이 포함되었는가?
- [ ] 마지막에 정리(cleanup) SQL이 포함되었는가?
