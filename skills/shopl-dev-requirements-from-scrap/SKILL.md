## 목적

Scrap된 기획 자료를 그대로 요약하지 않고, **현재 백엔드 구현과 대조하여 실제 변경이 필요한 백엔드 요구사항**으로 재구성함.

주요 산출물:

- 현재 구현 상태 정리
- 기획 요구사항 요약
- 현재 구현 대비 Gap 분석
- 화면 단위 백엔드 변경 요구사항
- API/DB/계산/Excel/검증 영향 범위
- 기획/정책 확인 필요 항목
- 향후 백엔드 구현용 화면 단위/공통 로직 단위 하위작업 이름

최종 결과에서 반드시 구분할 항목:

- 이미 구현된 기능
- 부분 구현된 기능
- 신규 구현 필요 기능
- 현재 구현과 기획이 어긋나는 기능
- 기획/정책 확인 필요 항목
- 프론트 중심 가능성이 높은 항목

## 사용 시점

사용자가 다음과 같이 요청하면 이 스킬 사용.

- “scrap된 기획서에서 백엔드 요구사항 뽑아줘”
- “Figma scrap 보고 백엔드 요구사항 정리해줘”
- “현재 구현 기준으로 기획서를 어떻게 바꿔야 하는지 정리해줘”
- “기획서와 구현을 비교해서 gap 분석해줘”
- “스크린샷/와이어프레임/엑셀 샘플을 보고 서버 작업 목록 뽑아줘”
- “이번 스프린트에서 백엔드가 실제로 해야 할 일 정리해줘”

## 필수 입력 계약

이 스킬은 scrap 자료와 현재 구현을 대조하는 스킬이므로, **scrap 자료 경로와 구현 확인 범위가 모두 있어야 실행 가능**함.

### 1. 필수 입력

다음 입력을 모두 받아야 함.

- Scrap 자료 경로
  - 예: `docs/<feature>/scrap`
  - 예: `docs/<feature>/scrap/index.md`
  - 예: `docs/<feature>/scrap/02_requirements/README.md`
- 구현 확인 범위
  - 특정 클래스 파일
  - 특정 패키지 디렉터리
  - 특정 모듈
  - 특정 API/controller/service 진입점
  - 사용자가 명시한 기능 흐름

### 2. 허용 가능한 구현 범위 예

특정 클래스:

- `core/src/main/kotlin/.../IoAttAggregationReportSet.kt`
- `api/src/main/kotlin/.../AttAggregationExcelDownloadService.kt`

특정 패키지:

- `api/src/main/kotlin/com/planetory/io/feature/export/service`
- `api/src/main/kotlin/com/planetory/io/feature/att_aggregation`

특정 모듈:

- `api`
- `core`
- `shopl-api/api-attendance`
- `shopl-core/core-system-feature`

기능 진입점:

- “근태 종합 리포트 다운로드 API”
- “Excel 생성 서비스”
- “리포트 설정 저장/수정 API”

### 3. 범위 미지정 시 동작

구현 확인 범위가 지정되지 않은 경우 다음 규칙 적용.

- 분석 시작 금지
- 전체 코드베이스 탐색 금지
- 임의 관련 패키지 추정 금지
- 백엔드 요구사항 문서 작성 금지
- 현재 구현 대비 Gap 분석 금지
- 구현 하위작업 목록 작성 금지
- 먼저 사용자에게 구현 확인 범위 지정 요청

사용자에게 요청할 질문 예:

```text
현재 구현과 대조할 클래스/패키지/모듈 범위를 지정해주세요.
예: `AttAggregationExcelDownloadService`, `IoAttAggregationReportSet`, `api/.../export/service`
```

### 4. 범위 확장 제한

지정된 범위 밖의 코드는 다음 경우에만 추가 확인 가능.

- 지정 클래스가 직접 import하거나 호출하는 경우
- DTO/Entity/Mapper/Validator/Repository처럼 기능 흐름상 필수 연결인 경우
- 지정 패키지 내부 동작을 이해하는 데 필요한 인접 파일인 경우
- 사용자가 추가 범위 확인을 승인한 경우

범위 확장이 필요하면 다음 형식으로 먼저 설명 후 진행.

```md
추가 확인이 필요한 범위:

- `path/to/AdditionalFile.kt`
- 사유: 다운로드 요청 DTO가 해당 서비스의 필수 입력값이므로 검증 기준 확인 필요
```

### 5. 최종 문서 분석 범위 기록

최종 요구사항 문서에는 반드시 “분석 범위”를 기록.

예:

```md
## 분석 범위

### Scrap 자료

- [Scrap index](./scrap/index.md)
- [요구사항 README](./scrap/02_requirements/README.md)

### 현재 구현 확인 범위

- `core/src/main/kotlin/.../IoAttAggregationReportSet.kt`
- `api/src/main/kotlin/.../AttAggregationExcelDownloadService.kt`

### 범위 제외

- 출퇴근 다운로드 구현은 이번 분석 범위 제외
- 프론트 라우팅 구현은 이번 분석 범위 제외
```

## 핵심 원칙

### 1. 기획서 원문 전사 금지

Scrap 문서나 화면에 있는 내용을 그대로 “신규 요구사항”으로 쓰지 않음.

현재 구현 확인 후 다음 중 하나로 분류 필요.

- 현재 구현 완료
- 현재 구현 일부 존재
- 데이터 모델만 존재
- API 요청값만 존재
- 계산 로직만 존재
- Excel 출력만 미구현
- 신규 구현 필요
- 기획 확인 필요
- 프론트 중심 가능

### 2. 변경 필요사항 중심 작성

요구사항은 “기능 설명”이 아니라 **현재 구현 대비 변경 필요사항** 중심으로 작성.

나쁜 예:

- “사용자는 휴무유형별 휴일 근로시간을 확인할 수 있다.”

좋은 예:

- “현재 휴일 근로시간은 합산값만 산출되므로, 휴무유형별 산출값을 담는 도메인 구조와 요약/상세 Excel 동적 컬럼 추가 필요.”

### 3. 화면 요소와 백엔드 구현 요소 매핑

화면/기획 요소는 가능한 한 다음 구현 요소와 연결.

- Entity
- Request DTO
- Response DTO
- Mapper
- Validator
- Controller
- Service entrypoint
- Reader/Factory/Aggregator
- Repository/Query
- Excel drawer
- Column adjustor
- Message key
- Error code
- Migration/Flyway SQL

### 4. 불명확 항목 단정 금지 — 반드시 전용 섹션으로 집계

다음 항목은 단순 주석이나 각 화면 내부 확인 필요 항목으로만 끝내지 않고, **문서 내 전용 섹션(§미확인/모호/추정 항목)에 반드시 한곳에 모아 기록**.

- OCR로 식별이 애매한 화면 문구
- scrap 문서와 이미지가 충돌하는 내용
- 기획 문서와 샘플 Excel이 다른 내용
- 현재 구현과 기획 의도가 다른 내용
- 경계값이 명확하지 않은 정책
- 프론트 처리인지 백엔드 처리인지 불명확한 내용
- 저장형 설정인지 다운로드 1회성 옵션인지 불명확한 내용
- **스크랩에 명시되지 않아 내가 추정한 내용과 그 추정 근거**
- **스크랩에서 전달되지 않아 누락된 것 같다고 의심되는 항목**
- **내 Gap 분류(구현 일부/신규 필요 등) 중 확신이 부족한 분류**
- **사용자에게 직접 확인·질문해야 할 사항**

### 5. 근거 링크 필수화

요구사항 작성 시 가능한 한 scrap 자료와 구현 파일의 근거를 함께 명시.

링크 작성 기준:

- 요구사항 항목마다 관련 scrap 문서 링크 포함
- 같은 기능 디렉터리 내 문서끼리는 상대 경로 링크 사용
- 이미지/샘플 Excel도 가능한 경우 상대 경로 링크 사용
- 구현 근거는 코드 파일 경로를 backtick으로 명시
- 링크를 남길 수 없으면 “확인된 문서 없음” 또는 “스크린샷 OCR 기반 추정”으로 표시

## 입력 자료 수집 순서

### 1. 프로젝트 규칙 확인

작업 전 프로젝트 지침 확인.

- 최상위 `AGENTS.md`
- 작업 디렉터리에 더 가까운 `AGENTS.md`
- 문서 작성 규칙 또는 conventions 문서
- 필요한 경우 `docs-readability` 스킬

문서 작성 시 우선 적용할 규칙:

- H1 바로 아래 목차 포함
- 긴 줄글보다 목록/표 중심 구성
- 명사형 마무리 우선
- 파일명은 소문자 + 하이픈 선호
- 문서명에 날짜 포함 금지
- 확인되지 않은 내용은 추정 또는 확인 필요로 분리

### 2. Scrap 인덱스 확인

Scrap 디렉터리에서 우선 확인할 파일:

- `index.md`
- `README.md`
- `requirements.md`
- `02_requirements/README.md`
- `policy` 관련 문서
- `wireframes` 관련 문서

확인 순서:

1. 인덱스/README 계열로 전체 구조 파악
2. 요구사항 문서로 기능 후보 추출
3. 정책 문서로 경계값 확인

### 3. 주요 화면/이미지 확인

우선 확인할 이미지 유형:

## 분석 절차

### 1. 화면 단위 분해

Scrap의 와이어프레임과 정책서를 정확히 읽고, **작업 대상 화면을 먼저 리스트업**. 화면 단위로 요구사항을 묶으면 이후 개발설계 단계에서 재사용 매핑이 명확해짐.

화면 식별 기준:

- 사용자가 진입하는 고유한 화면(페이지, 모달, 탭, 뷰) 단위
- 각 화면의 목적, 사용자 동작, 필요 백엔드 데이터
- 여러 화면에서 공유하는 로직은 **공통 로직**으로 별도 식별

산출물 1 — 화면 목록:

```md
| 화면 | 백엔드 역할 | 사용하는 공통 로직 | 관련 scrap |
|---|---|---|---|
| 일간 메인 | 근무지별 집계 API | 조회 범위, 배치 근무지, 집계 기준 | [일간 화면](./scrap/01_policy/screenshot_policy_일간_집계_화면.png) |
| 월간 메인 | 매트릭스 API | 조회 범위, 배치 근무지 | [월간 화면](./scrap/01_policy/screenshot_policy_월간.png) |
| 구성원 모달 | 구성원 목록 API | 조회 범위, 권한 분기 | [권한 범위](./scrap/01_policy/screenshot_policy_일간_권한_범위.png) |
```

산출물 2 — 공통 로직 목록 (여러 화면이 공유하는 집계/권한/추출 로직):

```md
| 공통 로직 | 사용 화면 | 비고 |
|---|---|---|
| 조회 가능 근무지 범위 계산 | 전체 화면 | 역할별 분기 |
| 배치 근무지 결정 (구성원 추출) | 일간, 월간, 모달 | 중복 귀속 허용 |
| 컬럼별 집계 기준 근무지 분기 | 일간, 근무지 상세 | 컬럼마다 기준 상이 |
```

원칙:

- 화면 단위로 먼저 나눈 뒤, 공통 로직은 별도 섹션으로 분리.
- 공통 로직은 개별 화면 섹션에서 중복 작성하지 말고 참조.
- 하나의 화면이 여러 API/뷰 모드를 가질 수 있음 (예: 일간/월간 토글, 출퇴근뷰/스케줄뷰).

### 2. Scrap 요구사항 추출

화면 단위로 Scrap 자료에서 기능 후보를 추출.

기능 후보별 정리 항목:

- 소속 화면
- 기능명
- 사용자 동작
- 정책/경계값
- 입력값
- 출력값
- Excel 영향
- 관련 scrap 파일 링크
- 불명확한 문구

예시:

```md
| 화면 | 기능 후보 | 설명 | 관련 scrap | 비고 |
|---|---|---|---|---|
| 일간 메인 | 블록1 근무지 현황 | 출근전/근무중/근무종료/배정없음 | [집계 화면](./scrap/01_policy/screenshot_policy_일간_집계_화면.png) | 상태 판정 조건 확인 |
| 일간 메인 | 페이지네이션 | 50/100/150행 | [요구사항](./scrap/02_requirements/README.md) | 행 수 경계 확인 |
```

### 3. 현재 구현 매핑

각 기능 후보를 현재 구현 파일에 매핑.

매핑 기준:

- Entity 필드 존재 여부
- Request DTO 수신 여부
- Response DTO 노출 여부
- Mapper 변환 여부
- Validator 검증 여부
- Service 사용 여부
- 계산 로직 반영 여부
- Excel 헤더 반영 여부
- Excel 데이터 반영 여부
- 테스트 또는 샘플 존재 여부

예시:

```md
| 기능 | Entity | DTO | Validator | 계산 | Excel 출력 | 판단 |
|---|---|---|---|---|---|---|
| 야간근로 시간대 | 있음 | 있음 | 있음 | 있음 | 있음 | 기존 구현 존재 |
| 휴무유형별 휴일근로 | dayOffIds만 있음 | 일부 | 없음 | 합산만 있음 | 유형별 없음 | 신규 변경 필요 |
```

### 4. Gap 분류

각 기능을 다음 상태 중 하나로 분류.

| 상태 | 의미 |
|---|---|
| 구현 완료 | 기획 요구와 현재 구현이 대체로 일치 |
| 구현 일부 존재 | 필드/요청/계산 중 일부만 존재 |
| 구현 보정 필요 | 기능은 있으나 정책/경계값/대상 기준이 다름 |
| 신규 구현 필요 | 현재 구현 기반이 거의 없음 |
| 기획 확인 필요 | 요구사항 자체가 불명확하거나 충돌 |
| 프론트 중심 가능 | 백엔드 변경이 없거나 제한적일 수 있음 |

### 5. 백엔드 변경 요구사항 재서술

Gap 분석 결과를 바탕으로 백엔드 변경 요구사항 작성. **화면 단위 + 공통 로직 단위로 구성**.

작성 기준:

- "현재 상태"와 "변경 필요" 분리
- scrap 근거와 구현 근거 함께 명시
- 화면 섹션은 해당 화면 전용 요구사항만, 공통 로직은 별도 섹션
- API/DB/계산/Excel/검증 영향 범위 구분
- 기획 확인 필요 항목 별도 분리
- 이미 구현된 기능을 신규 구현처럼 작성하지 않음

### 6. 미확인/모호/추정 항목 컴파일 (필수)

전체 분석이 끝난 후, 반드시 **모든 불확실성·추정·의문사항을 한 곳에 모아** 별도 섹션으로 정리.

컴파일 대상:

- 스크랩/스크린샷 OCR에서 애매하게 읽힌 내용
- 문서 간 충돌 (정책서 vs 와이어프레임 vs 샘플 Excel)
- 정책에 경계값이 명시되지 않은 항목
- 프론트 vs 백엔드 처리 구분이 모호한 항목
- **스크랩에 명시되지 않아 내가 추정한 내용과 그 추정의 근거**
- **스크랩에서 빠졌을 가능성이 있다고 의심되는 항목**
- **내 Gap 분류(구현 일부/신규 필요 등) 중 확신이 부족한 분류와 그 이유**
- **사용자에게 확인·질문해야 할 사항**

### 7. 구현 하위작업 이름 추출

문서 마지막에는 백엔드 구현용 하위작업 이름을 화면 단위/공통 로직 단위로 추출.

작성 기준:

- 문서 작성 단계가 아니라 구현 작업 단위로 작성
- 동사형 또는 명사형 작업명 사용
- 큰 작업은 API/Domain/Excel/Validation으로 분리
- 테스트 작업도 별도 포함
- 작업명만 요청받은 경우 설명 없이 이름 목록 중심 작성

예시:

```md
1. 다운로드 기간 제한 검증 구현
2. 미래 날짜 다운로드 차단 구현
3. 공통 대상자 후보 추출 로직 정리
4. 기록 존재 사용자 필터 구현
5. 휴무유형별 계산 결과 도메인 모델 추가
6. 요약 시트 휴무유형별 동적 헤더 구현
7. 상세 시트 휴무유형별 동적 헤더 구현
8. 주 단위 기간 분할 로직 구현
9. 주간 요약 시트 생성 로직 구현
10. 대상자 필터 회귀 테스트 케이스 작성
```

### 8. 문서 출력 전 최종 검토 (필수)

백엔드 요구사항 문서를 최종 출력하기 전에 반드시 다음 두 가지를 확인.

1. 문서 내에 "미확인/모호/추정 항목" 섹션이 존재하는가?
2. 해당 섹션이 비어 있지 않은가?
   - 비어 있다면, 정말 모든 불확실성이 해소된 것인지 점검.
   - 추정한 내용이 하나도 없다면 "식별된 미확인 항목 없음. 단, 문서에 명시된 내용만으로 모든 판단이 가능했음"이라고 서술.
   - DL을 false로 설정하는 구조가 불가능하다면 추정 항목이 하나도 없음을 단정하지 말고 다시 점검.

## 개발설계 단계로의 인계

이 스킬의 산출물(요구사항 문서)은 다음 단계인 **개발설계(시스템 구조 설계, API 설계)** 의 입력. 화면 단위로 구성된 요구사항은 아래 원칙으로 재사용 매핑.

### 1. 공통 로직 → Shared 컴포넌트 추출

여러 화면이 공유하는 로직은 중복 구현을 피하기 위해 Shared Service / Facade / Query Repository로 추출.

| 공통 로직 (요구사항 문서) | 설계 단계 추출 대상 |
|---|---|
| 조회 가능 근무지 범위 계산 | 권한 Facade / Query (예: `WorkplaceViewScopeFacade`) |
| 배치 근무지 결정 (구성원 추출) | Domain Service / Aggregator (예: `WorkplaceMemberAllocator`) |
| 컬럼별 집계 기준 근무지 분기 | 집계 전략 / Calculator (예: `WorkplaceAggregationPolicy`) |
| 권한 분기 (관리 vs 타 그룹) | 권한 컨텍스트 / Resolver |

원칙:

- 공통 로직은 단일 구현 지점에서 제공, 각 화면 Service가 주입받아 사용.
- 화면 Service에는 화면 전용 조립만 남김 (공통 로직 중복 구현 금지).

### 2. 화면별 → Controller / Service / DTO 분리

각 화면은 별도 진입점을 가짐. 공통 로직을 주입받아 화면 전용 응답을 조립.

```text
[화면]              [Controller]               [Service]                 [공통 로직]
일간 메인  →  WorkplaceDailyController  →  WorkplaceDailyService  →  WorkplaceViewScopeFacade
                                                            →  WorkplaceMemberAllocator
                                                            →  WorkplaceAggregationPolicy
월간 메인  →  WorkplaceMonthlyController →  WorkplaceMonthlyService →  (동일 공통 로직 주입)
구성원 모달 →  WorkplaceMemberController →  WorkplaceMemberService →  (동일 공통 로직 주입)
```

### 3. 재사용 매트릭스 작성

개발설계 문서에는 각 화면이 어떤 공통 컴포넌트를 사용하는지 매트릭스로 정리. 이 매트릭스는 요구사항 문서의 §1 화면 목록 + §2 공통 로직 목록에서 직접 도출.

```md
| 화면 | 조회 범위 | 배치 근무지 | 집계 기준 | 권한 분기 |
|---|---|---|---|---|
| 일간 메인 | ✓ | ✓ | ✓ | ✓ |
| 월간 메인 | ✓ | ✓ | — | ✓ |
| 구성원 모달 | ✓ | ✓ | — | ✓ |
| 근무지 상세 | ✓ | ✓ | ✓ | — |
```

### 4. 인계 시 확인 항목

요구사항 문서에서 개발설계로 넘길 때 반드시 확인:

- 공통 로직이 2개 이상 화면에서 사용되는지 (1개면 공통이 아닌 화면 전용으로 둘 것)
- 각 화면의 Request/Response DTO가 공통 모델(근무지 식별, 권한 컨텍스트, 날짜 범위)을 공유하는지
- 동일 집계 로직을 화면마다 복사하지 않고 단일 지점에서 제공하는지
- “개발 설계 단계에서 결정”으로 위임한 항목(성능, 연산 주체, 빈 결과 응답 형태)이 이 단계에서 설계됐는지

## 흔한 시행착오 및 회피 원칙

스크랩 기획서로부터 백엔드 요구사항을 구체화할 때 반복적으로 발생하는 시행착오와 회피 원칙.

### 1. 스크랩과 원본 저장소의 관계 오해

**시행착오**

스크랩된 정책서와 `shopl-pm` 같은 원본 저장소를 서로 다른 신뢰도 계층으로 오해함. 결과적으로 스크랩(와이어프레임 + 정책서)을 정확히 읽지 않은 채 원본 저장소만 우선 탐색하거나, 이미 스크랩 안에 답이 있는 항목을 “확인 필요”로 과다 분류함.

**사실 관계**

스크랩된 기획서에 첨부된 정책서는 피그마가 마크다운 원본을 렌더링한 결과를 캡처한 것이며, 그 마크다운 원본 링크는 `shopl-pm` 저장소를 바라보고 있어 **내용이 완전히 동일**함. 스크랩 소스의 와이어프레임과 정책서는 모두 정확한 자료임.

**회피 원칙**

- 스크랩의 정책서와 와이어프레임을 먼저 정확하게 읽고 분석할 것. 스크랩이 부정확해서 원본을 찾는 것이 아님.
- `shopl-pm` 원본을 별도로 확인하는 의미는 텍스트 원본 접근(복사/추출/전체 탐색 용이)이지, 신뢰도 차이가 아님.
- 텍스트 추출이 필요한 경우(표 복사, 조건문 전문 확보, 전역 검색)에만 원본으로 가고, 시각 확인은 스크랩으로 충분.
- 스크랩 안에 답이 있음에도 “확인 필요”로 넘기지 말 것.

### 2. 리뷰 문서와 요구사항 문서 이중 유지

**시행착오**

같은 기능에 대해 기획 검토용 리뷰 문서와 백엔드 요구사항 문서를 따로 만들어 병행 유지함. 결정이 바뀔 때마다 양쪽을 동기화해야 하는 비용이 발생하고, 한쪽만 갱신되는 정합성 오류가 반복됨.

**회피 원칙**

- 같은 주제의 산출 문서는 하나로 유지할 것.
- 기획 검토 메모(사이드 이펙트, 기술 타당성, 결정 내역)는 요구사항 문서의 한 섹션(예: “기획 검토 결정 사항”)으로 흡수.
- 검토용 임시 메모는 본문에 편입한 뒤 삭제. 단일 진실 소스.

### 3. 외부 링크로 본문을 비움

**시행착오**

판단에 필요한 집계 기준표, 산출 조건, 분기 규칙을 “`policy §6` 참조”, “§7.8 따름” 식으로 링크만 걸어둠. 문서를 읽는 사람이 원본을 오가야 해서 가독성이 떨어짐.

**회피 원칙**

- 판단에 필요한 내용은 본문에 인라인으로 풀어 작성할 것. 집계 기준표, 컬럼별 산출 조건, 분기 규칙은 표로 직접 작성.
- 외부 링크(`shopl-pm`, scrap)는 출처 표시용(“관련 자료” 섹션)으로만 남김.
- 읽는 사람이 문서 하나로 결론에 도달할 수 있어야 함.

### 4. 상태 라벨로 문서를 칸반화

**시행착오**

항목마다 `[해소]/[잔여]/[부분 해소]` 같은 상태 라벨을 붙여 추적표처럼 만듦. 라벨이 많아지면 본문 서술이 표 위주로 압축되어 가독성이 떨어짐.

**회피 원칙**

- 상태는 라벨이 아니라 서술로 표현할 것. “정의됨”, “추가 정의 필요”, “기획 재정의 예정” 식의 문장으로 작성.
- 문서는 산출물이지 진행 추적 도구가 아님. 상태 추적은 Jira 등 별도 도구로.

### 5. 추상 도메인 용어로 소통

**시행착오**

“다축 병합 집계 아키텍처”, “담당 근무지 집계 귀속” 같은 추상 용어를 그대로 질문/검토에 사용함. 사용자가 의도를 파악하지 못해 매번 되물어야 함.

**회피 원칙**

- 도메인 용어라도 구체 사례로 환원할 것.
  - 나쁜 예: “다축 병합 집계 아키텍처를 어떻게 할까요?”
  - 좋은 예: “한 직원이 A근무지 스케줄 배정 + B근무지 실제 출근 → 양쪽 근무지에 카운트. 이 병합을 단일 쿼리에서 할까요, 분리 후 메모리 병합할까요?”
- 확인받을 때는 “이렇게 이해하면 맞나?” 형태로 예시를 제시.

### 6. 기획 검토와 개발 설계 범위 혼동

**시행착오**

성능 대책, 캐시, 연산 주체, 아키텍처, 응답 포맷 등 개발 설계 항목을 기획 검토 단계에서 끌어들임. 검토가 끝나지 않고 “확인 필요”가 과다 발생함.

**회피 원칙**

- 기획 검토는 “무엇을 만들까”에 집중. “어떻게 만들까”는 개발 설계 단계로 위임.
- 개발 설계로 위임하는 대표 항목: 성능(캐시/사전 집계/실시간), 연산 주체(서버/클라이언트), 아키텍처(단일 쿼리/메모리 병합), 빈 결과 응답 형태.
- 이들 항목은 “실제 개발 설계 단계에서 결정”으로 명시적으로 범위 밖 처리.
- 프론트 영역(색상 강도, 보기 설정 영속화, UI 상태)도 마찬가지로 명시적 분리.

### 7. 산출물 위치 오배치

**시행착오**

요구사항 문서를 `scrap/02_requirements/` 안에 넣음. scrap은 입력 자료인데 산출물이 같이 있어 카테고리 경계가 흐려짐.

**회피 원칙**

- scrap(입력 자료)과 산출 문서는 물리적으로 분리할 것.
- 요구사항 문서는 기능 최상위(`docs/<feature>/`)에, scrap은 `docs/<feature>/scrap/`에.
- 위치가 틀리면 산출물인지 입력 자료인지 의미가 틀려짐.
