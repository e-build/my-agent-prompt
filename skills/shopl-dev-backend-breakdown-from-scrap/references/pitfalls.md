# 흔한 시행착오 및 회피 원칙

스크랩 기획서로부터 백엔드 구현 분해 항목을 구체화할 때 반복적으로 발생하는 시행착오. 분석 중 판단이 애매할 때 참조.

## 1. 스크랩과 원본 저장소의 관계 오해

**시행착오**: 스크랩된 정책서와 `shopl-pm` 같은 원본 저장소를 서로 다른 신뢰도 계층으로 오해함. 결과적으로 스크랩(와이어프레임 + 정책서)을 정확히 읽지 않은 채 원본 저장소만 우선 탐색하거나, 이미 스크랩 안에 답이 있는 항목을 "확인 필요"로 과다 분류함.

**사실 관계**: 스크랩된 기획서에 첨부된 정책서는 피그마가 마크다운 원본을 렌더링한 결과를 캡처한 것이며, 그 마크다운 원본 링크는 `shopl-pm` 저장소를 바라보고 있어 **내용이 완전히 동일**함. 스크랩 소스의 와이어프레임과 정책서는 모두 정확한 자료임.

**회피 원칙**:
- 스크랩의 정책서와 와이어프레임을 먼저 정확하게 읽고 분석할 것. 스크랩이 부정확해서 원본을 찾는 것이 아님.
- `shopl-pm` 원본을 별도로 확인하는 의미는 텍스트 원본 접근(복사/추출/전체 탐색 용이)이지, 신뢰도 차이가 아님.
- 텍스트 추출이 필요한 경우(표 복사, 조건문 전문 확보, 전역 검색)에만 원본으로 가고, 시각 확인은 스크랩으로 충분.
- 스크랩 안에 답이 있음에도 "확인 필요"로 넘기지 말 것.

## 2. 리뷰 문서와 구현 분해 문서 이중 유지

**시행착오**: 같은 기능에 대해 기획 검토용 리뷰 문서와 백엔드 구현 분해 문서를 따로 만들어 병행 유지함. 결정이 바뀔 때마다 양쪽을 동기화해야 하는 비용이 발생하고, 한쪽만 갱신되는 정합성 오류가 반복됨.

**회피 원칙**:
- 같은 주제의 산출 문서는 하나로 유지할 것.
- 기획 검토 메모(사이드 이펙트, 기술 타당성, 결정 내역)는 구현 분해 문서의 한 섹션(예: "기획 검토 결정 사항")으로 흡수.
- 검토용 임시 메모는 본문에 편입한 뒤 삭제. 단일 진실 소스.

## 3. 외부 링크로 본문을 비움

**시행착오**: 판단에 필요한 집계 기준표, 산출 조건, 분기 규칙을 "`policy §6` 참조", "§7.8 따름" 식으로 링크만 걸어둠. 문서를 읽는 사람이 원본을 오가야 해서 가독성이 떨어짐.

**회피 원칙**:
- 판단에 필요한 내용은 본문에 인라인으로 풀어 작성할 것. 집계 기준표, 컬럼별 산출 조건, 분기 규칙은 표로 직접 작성.
- 외부 링크(`shopl-pm`, scrap)는 출처 표시용("관련 자료" 섹션)으로만 남김.
- 읽는 사람이 문서 하나로 결론에 도달할 수 있어야 함.

## 4. 상태 라벨로 문서를 칸반화

**시행착오**: 항목마다 `[해소]/[잔여]/[부분 해소]` 같은 상태 라벨을 붙여 추적표처럼 만듦. 라벨이 많아지면 본문 서술이 표 위주로 압축되어 가독성이 떨어짐.

**회피 원칙**:
- 상태는 라벨이 아니라 서술로 표현할 것. "정의됨", "추가 정의 필요", "기획 재정의 예정" 식의 문장으로 작성.
- 문서는 산출물이지 진행 추적 도구가 아님. 상태 추적은 Jira 등 별도 도구로.

## 5. 추상 도메인 용어로 소통

**시행착오**: "다축 병렬 집계 아키텍처", "담당 근무지 집계 귀속" 같은 추상 용어를 그대로 질문/검토에 사용함. 사용자가 의도를 파악하지 못해 매번 되물어야 함.

**회피 원칙**:
- 도메인 용어라도 구체 사례로 환원할 것.
  - 나쁜 예: "다축 병렬 집계 아키텍처를 어떻게 할까요?"
  - 좋은 예: "한 직원이 A근무지 스케줄 배정 + B근무지 실제 출근 → 양쪽 근무지에 카운트. 이 병합을 단일 쿼리에서 할까요, 분리 후 메모리 병합할까요?"
- 확인받을 때는 "이렇게 이해하면 맞나?" 형태로 예시를 제시.

## 6. 기획 검토와 개발 설계 범위 혼동

**시행착오**: 성능 대책, 캐시, 연산 주체, 아키텍처, 응답 포맷 등 개발 설계 항목을 기획 검토 단계에서 끌어들임. 검토가 끝나지 않고 "확인 필요"가 과다 발생함.

**회피 원칙**:
- 기획 검토는 "무엇을 만들까"에 집중. "어떻게 만들까"는 개발 설계 단계로 위임.
- 개발 설계로 위임하는 대표 항목: 성능(캐시/사전 집계/실시간), 연산 주체(서버/클라이언트), 아키텍처(단일 쿼리/메모리 병합), 빈 결과 응답 형태.
- 이들 항목은 "실제 개발 설계 단계에서 결정"으로 명시적으로 범위 밖 처리.
- 프론트 영역(색상 강도, 보기 설정 영속화, UI 상태)도 마찬가지로 명시적 분리.

## 7. 산출물 위치 오배치

**시행착오**: 구현 분해 문서를 `scrap/02_requirements/` 안에 넣음. scrap은 입력 자료인데 산출물이 같이 있어 카테고리 경계가 흐려짐.

**회피 원칙**:
- scrap(입력 자료)과 산출 문서는 물리적으로 분리할 것.
- 구현 분해 문서는 기능 최상위(`docs/<feature>/`)에, scrap은 `docs/<feature>/scrap/`에.
- 위치가 틀리면 산출물인지 입력 자료인지 의미가 틀려짐.
