## 엔드포인트 상세

### 1. API 이름

**METHOD** `/api/v1/...`

목적:

- 이 API가 담당하는 책임

권한:

- 누가 호출 가능한지

요청 개념:

- 필요한 입력값 수준 정리
- 세부 DTO 필드는 필요 시 별도 정리

응답 개념:

- 어떤 정보가 돌아와야 하는지
- 성공 시 클라이언트가 무엇을 할 수 있어야 하는지

비고:

- 후속 액션
- 제약사항

### 2. API 이름

**METHOD** `/api/v1/.../{id}`

목적:

- 이 API가 담당하는 책임

권한:

- 누가 호출 가능한지

요청 개념:

- 필요한 입력값 수준 정리

응답 개념:

- 반환 개념 정리

## 상태 / 예외 시나리오

### 상태 분류

- 정상 상태
- 대상 없음
- 권한 부족
- 이미 처리됨
- 제한 초과
- 만료됨
- 삭제됨

### 예외 시나리오 정리

- 시나리오 1
  - HTTP status
  - 에러 코드
  - 클라이언트 후속 액션
- 시나리오 2
  - HTTP status
  - 에러 코드
  - 클라이언트 후속 액션

## 에러 코드

> 원칙: 에러 코드는 API 정의 시점에 정의

| 코드 | HTTP Status | 의미 | 사용자 메시지 |
|---|---|---|---|
| `FEATURE_SAMPLE_ERROR` | `400` | 설명 | 사용자 문구 |
| `FEATURE_NOT_FOUND` | `404` | 설명 | 사용자 문구 |
| `FEATURE_FORBIDDEN` | `403` | 설명 | 사용자 문구 |

## 에러 코드별 클라이언트 처리 기준

| 코드 | 권장 1차 처리 | Primary CTA | 이동 대상 | 비고 |
|---|---|---|---|---|
| `FEATURE_SAMPLE_ERROR` | 인라인 오류 또는 안내 | 다시 시도 | 현재 화면 | 설명 |
| `FEATURE_NOT_FOUND` | 대상 없음 안내 | 목록으로 이동 | 목록 화면 | 설명 |
| `FEATURE_FORBIDDEN` | 권한 안내 | 확인 | 현재 화면 또는 이전 화면 | 설명 |

## 응답 예시 메모

### 성공 예시

```json
{
  "result": "SUCCESS",
  "data": {},
  "error": null
}
```

### 실패 예시

```json
{
  "result": "ERROR",
  "data": null,
  "error": {
    "code": "FEATURE_SAMPLE_ERROR",
    "message": "사용자에게 보여줄 문장"
  }
}
```

## 구현 전 체크포인트

- 기능 문서와 API 문서가 일치하는지
- 상태 분기가 클라이언트 구현 가능 수준인지
- 에러 코드가 충분히 구분 가능한지
- 권한 정책이 모호하지 않은지
- 구현 중 새 정책이 생기면 문서 먼저 갱신할지 확인
