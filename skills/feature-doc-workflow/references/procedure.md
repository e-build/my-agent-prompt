- 각 단계별 목적, 우선 확인 문서, 산출물, 완료 기준을 정리
- SKILL.md의 6단계 요약을 상세화
- 단계 진입 전 체크리스트로 게이트 역할 수행

## 공통 전제

- 모든 기능 작업은 기본적으로 아래 6단계 순서를 따름
- 이전 단계 합의 없이 다음 단계로 건너뛰지 않음
- 기능 기획 문서와 기술 문서는 분리
- 같은 내용을 여러 문서에 중복 정의하지 않음
- 문서가 코드보다 늦게 가면 안 됨 — 구현 전 최소한 API와 정책 합의 선행

## 1단계 요구사항 수집 및 정의

### 목적

- 무엇을 만들지, 왜 만드는지 명확히 정의
- 사용자 시나리오, 기능 범위 고정
- 정책 결정보다 **무엇이 필요한지**에 집중

### 우선 확인 문서

- 루트 `PROGRESS.md`
- 루트 진입 문서 (`docs/index.md` 등)
- 제품 요구사항 / 로드맵 문서
- 관련 기능의 기존 `docs/features/{feature}/v{version}/` 문서
- 기존 관련 기능의 API / 화면정의서 (영향 범위 파악)

### 산출물

- `docs/features/{feature}/v{version}/01-requirements.md`

### 완료 기준

- 기능 목표 명확
- 사용자 시나리오 정리 완료
- 포함 범위 / 제외 범위 명확
- DB 스키마 / 코드 설계로 내려가지 않은 상태 유지
- 기존 기능과의 정합성 충돌 확인

### 시작 전 체크

- [ ] 관련 문서 위치 확인
- [ ] 기존 유사 기능 문서 존재 여부 확인
- [ ] 현재 기능이 제품 범위(요구사항/로드맵)에 부합하는지 확인
- [ ] 이미 구현된 기능(API/화면정의서)과의 정합성 충돌 사전 확인

## 2단계 기획 및 정책 수립

### 목적

- 요구사항을 바탕으로 실제 처리 규칙과 UX 흐름 결정
- 정책, 예외 상황, 액션 기준 합의
- **어떻게 처리할지**에 집중

### 우선 확인 문서

- 루트 `PROGRESS.md`
- 해당 기능의 `01-requirements.md`
- 제품 요구사항 / 로드맵

### 산출물

- `docs/features/{feature}/v{version}/02-plans.md`

### 완료 기준

- 정책 및 예외 상황 정리 완료
- UX 흐름 정리 완료
- 오픈 포인트 명시
- 요구사항과 정책 간 모순 없음 확인

### 시작 전 체크

- [ ] 요구사항 문서 최신 상태 확인 (`01-requirements.md`)
- [ ] 요구사항과 정책 간 모순 없는지 교차 확인

## 3단계 화면정의서

### 목적

- 화면 단위로 사용자 흐름과 상태, CTA, 에러 분기를 정리
- 클라이언트 구현의 기준이 되는 화면 단위 산출물 확보

### 우선 확인 문서

- 해당 기능의 `02-plans.md`
- `docs/screen-definitions.md` (화면 ID 원문)
- 디자인 시스템 / 토큰 / 패턴 문서 (있으면)

### 산출물

- `docs/features/{feature}/v{version}/03-screens.md`
- 화면 ID는 루트 `docs/screen-definitions.md`에도 등록

### 완료 기준

- 화면 ID / 사용자 목적 / Primary·Secondary CTA 정의
- 상태 분기 (기본 / Empty / Loading / Error / Success) 정리
- 에러 코드별 CTA 매핑 확인
- 화면 시안의 오류/상태 예시는 실제 정책과 구분 명확 (예시 vs 합의 정책)

### 시작 전 체크

- [ ] 기획/정책 문서 최신 상태 확인 (`02-plans.md`)
- [ ] 화면 ID 체계와 기존 화면정의서 일관성 확인
- [ ] 화면에 등장하는 에러 코드가 아직 정의 전이면 표기만 하고 4단계에서 확정

## 4단계 API 명세 정의

### 목적

- 클라이언트와 서버가 합의할 인터페이스 정의
- 요청/응답 책임, 상태 구분, 액션 단위 명확화
- 해당 API에서 발생 가능한 에러를 인터페이스 수준에서 정의

### 우선 확인 문서

- 루트 `PROGRESS.md`
- 해당 기능의 `02-plans.md`
- 해당 기능의 화면정의서 (`03-screens.md`)
- `docs/engineering/api-conventions.md`
- 관련 기존 API 문서

### 원칙

- **에러 코드는 API 정의 시점에 함께 정의**
- 기능 정의 단계에서 에러 코드 상세화 금지
- 구현 단계로 미루지 않음

### 산출물

- `docs/features/{feature}/v{version}/04-api.md` (에러 코드 섹션 포함)
- 필요 시 공통 에러 규약 문서 보강

### 완료 기준

- 엔드포인트 책임 정의
- 주요 요청/응답 개념 정의
- 상태/예외 시나리오 정리
- 공통 응답 포맷 준수
- 정상 / 비정상 상태 구분 가능
- 클라이언트 분기 가능한 수준의 에러 코드 정의 완료
- 사용자 노출 메시지 기준 일관성 확보

### 시작 전 체크

- [ ] 기획/정책 문서 최신 상태 확인 (`02-plans.md`)
- [ ] 화면정의서에 표기된 상태 / 에러 분기가 API에서 커버 가능한지 확인
- [ ] 공통 응답 규약 확인
- [ ] 인증/인가 영향 범위 확인

## 5단계 백엔드 시스템 설계

### 목적

- API와 정책을 바탕으로 내부 구조 설계
- 도메인, 계층, 책임 분리 방식 정리
- 기획, API, 설계 간 충돌 확인 (검토 포함)

### 우선 확인 문서

- 루트 `PROGRESS.md`
- 관련 `docs/features/{feature}/v{version}/` 문서
- 관련 화면정의서 (`03-screens.md`)
- 관련 `docs/engineering/` 공통 문서
- 시스템 아키텍처 문서 (있으면)

### 산출물

- 필요 시 `docs/features/{feature}/v{version}/05-design.md`
- 구현 계획 또는 설계 메모

### 완료 기준

- 주요 컴포넌트 책임 정리
- 도메인 경계 정리
- 테스트 포인트 식별 완료
- 요구사항과 API가 일치하는지 검토
- 권한 정의 모순 없는지 검토 (예: Owner / Admin / Member)
- 프런트/백엔드 분기 처리 가능한지 검토
- 공통 응답 규약과 충돌 없는지 검토

### 시작 전 체크

- [ ] API와 에러 코드 정의 완료 여부 확인
- [ ] 권한 정책 확정 여부 확인
- [ ] 구현 범위가 MVP와 맞는지 확인

## 6단계 구현 및 검증

### 목적

- 합의된 문서 기준으로 코드 반영
- 구현 결과가 문서와 일치하는지 검증

### 우선 확인 문서

- 기능 계획 문서
- 화면정의서
- API 문서
- 설계 문서
- 공통 응답 규약 문서

### 원칙

- 문서 합의 없는 임의 스펙 추가 금지
- 구현 중 새 정책이 나오면 문서 먼저 갱신
- 가능하면 작은 단위로 구현 및 검증 진행
- 구현 완료 주장 전 검증 필수 (테스트 없으면 완료 불가)

### 완료 기준

- 핵심 시나리오 검증 완료
- 예외 시나리오 검증 완료
- 응답 포맷 / 상태 코드 / 에러 코드 검증 완료
- 문서와 실제 동작 간 차이 없음

### 시작 전 체크

- [ ] 문서 합의 상태 확인
- [ ] 테스트 대상 시나리오 정리
- [ ] 변경 영향 범위 확인

## 단계 게이트 요약

각 단계는 다음 단계로 넘어가기 전 아래를 만족해야 한다.

| 게이트 | 통과 조건 |
|---|---|
| 1 → 2 | 요구사항 명확, 범위 고정, 기존 기능 충돌 확인 |
| 2 → 3 | 정책·예외·UX 흐름 합의, 오픈 포인트 명시 |
| 3 → 4 | 화면 상태/CTA/에러 분기 정리, 에러 코드 자리 표시 |
| 4 → 5 | 에러 코드 확정, 권한 정책 확정, 공통 규약 준수 |
| 5 → 6 | 설계 검토 완료, 테스트 포인트 식별 |
| 6 → 완료 | 핵심/예외 시나리오 검증, 문서-동작 일치 확인 |
