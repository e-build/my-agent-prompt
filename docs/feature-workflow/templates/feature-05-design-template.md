## 설계 범위

### 포함 범위

- 도메인 책임 구조
- 상태 모델
- 주요 흐름 설계
- 접근 제어 기준
- 테스트 포인트

### 제외 범위

- DB 스키마 상세
- DTO 세부 필드 전체 확정
- 인프라 구현 세부
- 코드 레벨 클래스 설계 전체

## 설계 원칙

- 상태와 책임을 클라이언트/서버 모두 이해 가능한 수준으로 명확화
- 정책 문서와 API 문서 간 불일치 방지
- 구현 복잡도보다 검증 가능성 우선
- 공통 규약과 충돌 없는 구조 유지

## 책임 분리

### 컨트롤러 계층

책임:

- 외부 요청 수신
- 응답 포맷 적용
- 서비스 호출

### 애플리케이션 서비스

책임:

- 유스케이스 오케스트레이션
- 상태 전이 처리
- 외부 도메인 협력

### 도메인 모델 / 도메인 서비스

책임:

- 핵심 비즈니스 규칙 보유
- 유효성 / 제한 / 전이 규칙 관리

### 인프라 계층

책임:

- 저장소, 외부 API, 메시징 등 구현

## 주요 흐름 설계

### 기본 성공 흐름

1. 요청 수신
2. 권한 확인
3. 대상 조회
4. 규칙 검증
5. 상태 변경 또는 결과 생성
6. 응답 반환

### 예외 흐름

1. 대상 없음
2. 권한 부족
3. 제한 초과
4. 중복 처리

| 생성 | 불가 | 가능 | 가능 | 설명 |
| 관리 액션 | 불가 | 제한 | 가능 | 설명 |

## 테스트 포인트

### 단위 테스트

- 정상 흐름
- 상태 전이
- 권한 검증
- 예외 처리

### 통합 테스트

- API 응답 포맷
- 인증/인가 분기
- 저장소 반영
- 에러 코드 검증

## 구현 전 결정 필요 항목

1. 상태 전이의 최종 규칙
2. 트랜잭션 경계
3. 외부 시스템 실패 시 보상/재시도 기준
4. 캐시 / 비동기 처리 필요 여부
